# AI 简历作品集网站 PRD

## 1. 文档信息

- 产品名称：AI 简历作品集网站
- 文档类型：产品需求文档
- 目标版本：MVP v1.0
- 主要场景：求职展示、面试讲解、个人项目作品集沉淀
- 核心形态：个人作品集网站 + Dify AI 简历助手 + SSE 流式对话 + 后台内容管理系统

## 2. 产品概述

AI 简历作品集网站是一个面向 HR、技术面试官和潜在合作方的个人展示型产品。它不只是静态简历页，也不是单纯的 AI 聊天 Demo，而是一个可以展示个人经历、项目证据、技术能力和 AI 应用开发能力的交互式作品集。

用户进入网站后，可以先通过首页快速了解候选人的岗位方向、核心技术栈和精选项目，也可以直接向 AI 简历助手提问，例如“你适合 AI 全栈岗位吗”“讲讲你的 Dify 项目”“你的前端能力体现在哪里”。AI 助手通过 Dify API 返回 SSE 流式回复，让访问者获得更自然、更高效的简历阅读体验。

## 3. 背景与问题

### 3.1 背景

传统个人简历通常以 PDF 或静态网页为主，信息是单向展示的。招聘方需要自己阅读、筛选和判断，很难快速追问项目细节、技术深度和岗位匹配度。

另一方面，如果只做一个 AI 聊天页面，又会缺少项目截图、视频、技术说明等可信证据，容易被理解成一个简单的聊天接口 Demo，无法充分证明候选人的真实项目经验。

因此，本产品需要把“AI 对话能力”和“作品集展示能力”结合起来，既让访问者可以交互式了解个人经历，也让页面本身具备可视化证据和专业表达。

### 3.2 需要解决的问题

- 静态简历缺少交互性，无法支持招聘方快速追问。
- 普通聊天 Demo 缺少项目证据，难以体现真实能力。
- 项目经历如果只用文字描述，说服力不足。
- 候选人的 AI 应用开发能力需要通过真实产品形态体现。
- HR 和面试官需要在短时间内判断候选人的岗位匹配度。

## 4. 产品目标

### 4.1 用户目标

- 让 HR 在 30 秒内理解候选人的岗位方向和核心能力。
- 让技术面试官快速查看项目截图、视频、技术栈和个人职责。
- 让访问者可以通过 AI 问答追问简历和项目细节。
- 让候选人在面试中可以直接打开网站辅助讲解项目。

### 4.2 产品目标

- 打造一个完整的个人作品集网站，而不是单一聊天页。
- 将 Dify AI 简历助手作为核心亮点放在首页。
- 使用 SSE 流式输出展示 AI 应用集成能力。
- 通过项目截图、视频和详情页增强可信度。
- 支持桌面端和移动端访问。

### 4.3 技术目标

- 前端固定使用 React 技术栈开发。
- 前端必须兼容桌面端和手机移动端。
- 后端固定使用 Python 技术栈开发。
- 数据库固定使用 MySQL。
- 系统需要支持项目图片、项目视频、简历文件等媒体资源的存储、引用和展示。
- 系统需要提供后台管理系统，用于项目、截图、视频、简历文件、站点基础信息和 AI 预设问题的增删改查。
- 后台管理系统需要登录保护，公开访客不能访问后台页面和管理接口。
- 前端支持流式文本渲染。
- Dify API Key 不暴露在前端。
- 支持前端请求后端代理接口，由后端转发到 Dify。
- 页面具备良好的响应式体验。
- MVP 版本结构简单，便于快速上线和迭代。

## 5. 目标用户与使用场景

### 5.1 目标用户

#### HR / 招聘负责人

- 目标：快速判断候选人是否符合岗位要求。
- 关注点：岗位方向、技术栈、项目经验、简历下载、联系方式。
- 典型行为：快速浏览首页、点击简历下载、查看项目标题和标签。

#### 技术面试官

- 目标：判断候选人的真实技术能力和项目深度。
- 关注点：项目架构、技术难点、接口实现、AI 应用落地、个人职责。
- 典型行为：查看项目详情、播放演示视频、向 AI 追问项目细节。

#### 潜在合作方

- 目标：了解候选人可以解决什么问题、能承接哪些类型的项目。
- 关注点：过往作品、技术能力、交付能力、联系方式。
- 典型行为：浏览项目集、查看 Demo、联系候选人。

#### 候选人本人

- 目标：沉淀个人作品，用于求职、面试、社交展示。
- 关注点：展示效果、可维护性、可扩展性、部署方便。
- 典型行为：持续补充项目截图、视频、项目说明和简历内容。

### 5.2 核心使用场景

#### 场景 1：HR 快速筛选

HR 打开网站后，在首屏看到候选人的姓名、目标岗位、核心技能标签和简历下载按钮。HR 可以快速判断是否继续查看，也可以点击预设问题，让 AI 简历助手总结候选人与岗位的匹配度。

#### 场景 2：技术面试官查看项目

技术面试官进入项目区，点击一个核心项目卡片，查看项目截图、演示视频、技术栈、个人职责和技术难点。对于不清楚的内容，可以返回 AI 助手继续提问。

#### 场景 3：候选人面试讲解

候选人在面试中打开网站，从项目卡片进入详情页，用截图、视频和架构说明辅助讲解项目，再通过 AI 助手演示交互式简历能力。

## 6. 产品定位

本产品定位为：

> 一个由 AI 简历助手驱动的个人作品集网站。

它需要同时满足三个层面的展示：

- 简历展示：让别人知道你是谁、能做什么、适合什么岗位。
- 项目展示：用截图、视频和说明证明你做过什么。
- 技术展示：用 Dify、SSE、前后端接口集成体现 AI 应用开发能力。

## 7. 信息架构

网站建议采用单页作品集结构，核心模块如下：

1. 首页首屏
2. AI 简历助手
3. 精选项目
4. 项目详情
5. 技术能力矩阵
6. 工作 / 教育 / 证书经历
7. 简历下载与联系方式

MVP 阶段可以采用单页滚动结构，项目详情使用弹窗或独立详情区。后续如果项目数量增多，再拆分为多页面。

## 8. 功能需求

### 8.1 首页首屏

#### 功能描述

首页首屏用于快速传达候选人的身份、方向和核心竞争力。

#### 展示内容

- 姓名
- 目标岗位
- 一句话定位
- 核心技术标签
- 简历下载按钮
- 联系方式入口
- AI 简历助手入口或首屏聊天框

#### 推荐文案方向

- AI 全栈开发工程师
- Dify / Agent / RAG 应用开发
- Vue / React / Node.js / Python / Three.js
- 用 AI 应用和前端工程能力解决真实业务问题

#### 验收标准

- 用户进入页面后，无需滚动即可看到候选人定位。
- 首屏包含至少 5 个核心技术标签。
- 首屏提供简历下载入口。
- 首屏能看到 AI 简历助手入口。

### 8.2 AI 简历助手

#### 功能描述

AI 简历助手是本产品的核心互动模块。用户可以通过自然语言提问，系统调用 Dify API，并以 SSE 流式方式展示回复。

#### 基础功能

- 支持用户输入文本问题。
- 支持点击预设问题快速提问。
- 支持 SSE 流式输出。
- 支持显示回答生成中状态。
- 支持连续对话上下文。
- 支持错误提示和重试。
- 支持清空当前对话。

#### 预设问题

- 请介绍一下你的核心项目。
- 你适合 AI 全栈岗位吗？
- 讲讲你的 Dify 简历助手项目。
- 你的前端能力体现在哪些项目里？
- 你的后端能力体现在哪些项目里？
- 你做过哪些 AI Agent 相关项目？
- 请用 1 分钟介绍你自己。

#### 状态设计

- 空状态：展示欢迎语和预设问题。
- 输入中：用户正在编辑问题。
- 请求中：已发送问题，等待 Dify 响应。
- 流式输出中：逐步展示 AI 回复内容。
- 完成状态：回答结束，输入框恢复可用。
- 错误状态：展示错误原因和重试按钮。
- 中断状态：用户可停止当前生成。

#### 验收标准

- 用户点击预设问题后，可以自动发送问题。
- 用户手动输入问题后，可以成功发送。
- AI 回复必须以流式方式逐步展示。
- 回复过程中输入框或发送按钮需要有合理禁用或状态提示。
- 网络错误时不能页面崩溃，需要显示可理解的错误信息。

### 8.3 精选项目展示

#### 功能描述

精选项目模块用于展示候选人的真实项目经历，是 AI 对话之外最重要的信任证据。

#### 项目数量

MVP 阶段建议展示 3 个项目，后续扩展到 5 个。

#### 推荐项目

- AI 简历助手
- AI-Agent 营销平台或智能仓储 AI-Agent 项目
- Three.js / 数字孪生项目
- Python / Node.js 后端项目
- 前端可视化或教学页面项目

#### 项目卡片字段

- 项目名称
- 项目封面图
- 一句话说明
- 技术栈标签
- 项目类型
- 个人职责摘要
- 查看详情按钮
- Demo / GitHub / 文档链接，按项目实际情况展示

#### 验收标准

- MVP 至少展示 3 个项目卡片。
- 每个项目至少包含 1 张封面图。
- 每个项目必须展示技术栈标签。
- 每个项目必须说明个人职责。
- 点击项目卡片或按钮可以查看详情。

### 8.4 项目详情

#### 功能描述

项目详情用于展示项目背景、功能截图、演示视频、技术架构和个人贡献。

#### 展示内容

- 项目背景
- 项目目标
- 主要功能
- 技术栈
- 项目截图
- 演示视频
- 技术架构说明
- 关键难点
- 解决方案
- 个人职责
- 项目成果

#### 截图与视频要求

- 普通项目：至少 1 张封面截图，建议 2-3 张功能截图。
- 核心项目：必须包含 30-60 秒演示视频。
- AI 项目：建议展示对话流式效果视频、Dify 工作流截图和最终效果截图。
- Three.js / 数字孪生项目：建议展示动态图或视频。
- 后端项目：建议展示接口调试截图、接口文档截图或架构图。

#### 验收标准

- 核心项目详情中可以看到视频或动态图。
- 项目详情需要明确写出“我负责什么”。
- 项目详情需要说明至少 1 个技术难点。
- 项目详情需要包含可视化证据，不能只有文字。

### 8.7 后台管理系统

#### 功能描述

后台管理系统用于维护公开作品集页面的数据和媒体资源。没有后台时，项目、截图、视频和简历文件只能通过手动改代码或手动改数据库维护，不利于长期迭代。因此后台管理系统纳入 MVP 范围。

#### 管理对象

- 项目：新增、编辑、删除、发布/隐藏、排序、设置精选项目。
- 项目详情：维护项目背景、目标、功能点、个人职责、技术难点、解决方案、技术栈和外部链接。
- 媒体资源：上传、查看、替换、删除图片、视频和 PDF 文件。
- 项目媒体关联：给项目设置封面图、截图、演示视频、架构图，并维护排序和说明文字。
- 简历文件：上传新版 PDF 简历，设置当前默认简历。
- 站点基础信息：维护姓名、岗位方向、一句话介绍、联系方式、GitHub/作品链接等基础配置。
- AI 预设问题：新增、编辑、删除、排序首页展示的预设问题。

#### 角色与权限

MVP 阶段只需要一个管理员角色。后台必须登录后才能访问。公开访客只能访问公开页面和公开 API，不能调用管理接口。

#### 后台页面建议

- 登录页
- 仪表盘首页
- 项目管理列表
- 项目新增/编辑页
- 媒体资源管理页
- 简历文件管理页
- 站点信息设置页
- AI 预设问题管理页

#### 验收标准

- 管理员可以登录和退出后台。
- 未登录访问后台页面时必须跳转到登录页。
- 管理员可以新增、编辑、删除项目。
- 管理员可以上传项目截图，并将截图关联到指定项目。
- 管理员可以上传 PDF 简历，并设置为当前简历。
- 管理员可以维护首页 AI 预设问题。
- 管理员修改项目或媒体后，公开作品集页面刷新即可看到最新内容。
- 删除项目和媒体默认使用软删除，避免误删导致公开页面资源失效。
- 后台接口不能暴露 Dify API Key、MySQL 密码或服务器真实文件路径。

## 9. 技术方案

### 9.1 前端技术选型

本项目前端固定使用 React 技术栈。

推荐方案：

- React + Vite
- TypeScript
- React Router
- CSS Modules、Tailwind CSS 或普通 CSS，具体在技术设计文档中确定
- fetch + ReadableStream 处理 SSE 流式响应

#### 选择 React 的原因

- 更适合展示通用前端能力。
- 对 AI 全栈岗位也有较高认可度。
- 与 Vite 结合可以快速完成 MVP。
- 适合组件化拆分聊天框、项目卡片、项目详情和媒体展示模块。
- 后续可以迁移到 Next.js，增强 SEO 和服务端渲染能力。

#### 移动端兼容要求

- 页面必须采用响应式布局。
- 手机宽度下首屏、聊天框、项目卡片、项目详情、视频播放都必须可正常浏览。
- 项目卡片在桌面端可以使用多列布局，在移动端应切换为单列布局。
- 聊天输入框在移动端不能被底部区域遮挡。
- 视频和图片必须设置最大宽度，避免横向滚动。
- 按钮、输入框和链接需要满足手机触控尺寸。

#### 暂不使用 UniApp 的原因

- 本项目主要访问场景是 Web 链接。
- HR 和面试官更习惯直接打开网页。
- UniApp 更适合小程序或 App 场景。
- MVP 阶段没有必要同时兼顾多端。

### 9.2 后端技术选型

本项目后端固定使用 Python + MySQL。

推荐方案：

- Python Web 框架：FastAPI，或 Flask
- 数据库：MySQL
- 数据访问：SQLAlchemy、SQLModel 或同类 ORM
- 文件上传：后端接收图片、视频和 PDF 简历文件
- 文件存储：MVP 可先使用服务器本地存储；后续可迁移到对象存储

后端主要职责：

- 代理 Dify API，保护 Dify API Key。
- 透传或转发 SSE 流式响应。
- 管理项目数据。
- 管理项目图片和视频资源。
- 管理简历文件下载。
- 提供前端需要的 REST API。

### 9.3 媒体资源存储方案

系统需要处理三类媒体资源：

- 项目图片：项目封面图、功能截图、架构图。
- 项目视频：核心项目演示视频、流式对话演示视频、Three.js / 数字孪生演示视频。
- 简历文件：PDF 简历。

MVP 推荐存储方式：

```text
后端接收文件上传
-> 校验文件类型和大小
-> 保存到服务器 uploads 目录
-> 在 MySQL 中保存文件元数据和访问 URL
-> 前端通过 URL 展示图片、播放视频或下载简历
```

MySQL 中至少需要保存：

- 文件 ID
- 文件类型，image / video / document
- 原始文件名
- 存储路径
- 访问 URL
- 文件大小
- MIME 类型
- 所属项目 ID，可为空
- 创建时间

媒体资源限制：

- 图片格式支持 jpg、jpeg、png、webp。
- 视频格式支持 mp4、webm。
- 简历文件支持 pdf。
- MVP 阶段不做在线视频转码。
- 视频文件应控制体积，建议单个视频不超过 50 MB。
- 上传接口需要限制文件类型和大小。

后续可扩展：

- 阿里云 OSS、腾讯云 COS、七牛云或 S3 兼容对象存储。
- 视频封面自动生成。
- 图片压缩和缩略图。
- CDN 加速。

### 9.4 后端代理方案

前端不应直接请求 Dify API，也不应在前端暴露 Dify API Key。

推荐请求链路：

```text
前端页面
-> 自定义后端代理接口
-> Dify API
-> Dify 返回 text/event-stream
-> 后端透传 SSE
-> 前端逐步渲染回复
```

### 9.5 接口设计

#### 发送聊天消息

```text
POST /api/chat
```

请求参数：

```json
{
  "message": "请介绍一下你的项目经历",
  "conversationId": "可选"
}
```

返回类型：

```text
text/event-stream
```

前端需要处理的事件：

- message：接收文本增量。
- done：回答完成。
- error：请求或生成失败。

#### 项目列表

```text
GET /api/projects
```

返回项目卡片需要的数据，包括项目名称、简介、技术栈、封面图和排序。

#### 项目详情

```text
GET /api/projects/{projectId}
```

返回项目详情，包括项目背景、功能说明、个人职责、技术难点、截图、视频和链接。

#### 媒体上传

```text
POST /api/media
```

用于上传项目图片、项目视频或简历文件。MVP 阶段可先作为管理端或本地维护接口，不一定对公开访问者开放。

#### 简历下载

```text
GET /api/resume/download
```

返回当前 PDF 简历文件。

### 9.6 Dify API 调用要求

请求 Dify 时需要使用 streaming 模式：

```json
{
  "inputs": {},
  "query": "用户问题",
  "response_mode": "streaming",
  "conversation_id": "可选",
  "user": "portfolio-visitor"
}
```

### 9.7 安全要求

- Dify API Key 只能保存在服务端环境变量。
- 前端代码不能出现 Dify API Key。
- 后端代理接口需要限制请求体大小。
- 媒体上传接口需要限制文件类型和文件大小。
- 媒体文件访问路径不能暴露服务器敏感目录。
- MySQL 连接信息只能保存在服务端环境变量。
- 可选：增加基础频率限制，避免接口被滥用。

## 11. 用户故事与验收标准

### Story 1：快速了解候选人

作为 HR，我希望打开网站后能快速看到候选人的岗位方向、技术栈和简历入口，以便判断是否继续了解。

验收标准：

- 首屏展示姓名、岗位方向和一句话定位。
- 首屏展示核心技术标签。
- 首屏提供简历下载按钮。
- 首屏可以进入 AI 简历助手。

### Story 2：通过 AI 追问简历

作为面试官，我希望可以直接询问候选人的项目经历和技术能力，以便快速获得结构化回答。

验收标准：

- 用户可以输入任意问题。
- 用户可以点击预设问题。
- AI 回复以流式方式展示。
- 对话失败时可以重试。
- 页面不会因为接口错误崩溃。

### Story 3：查看项目真实性证据

作为技术面试官，我希望查看项目截图、视频和技术说明，以便判断项目是否真实、候选人贡献是否明确。

验收标准：

- 每个项目有截图或封面图。
- 核心项目有视频或动态图。
- 项目详情说明个人职责。
- 项目详情说明技术难点和解决方案。

### Story 4：下载简历并联系候选人

作为招聘方，我希望在认可候选人后可以下载简历并获取联系方式，以便进入下一步沟通。

验收标准：

- 简历下载链接可用。
- 联系方式清晰展示。
- 移动端可以正常点击联系入口。

## 16. 风险与应对

### 风险 1：Dify API Key 泄露

应对方案：

- 使用后端代理接口。
- API Key 存放在服务端环境变量。
- 前端只请求自己的 `/api/chat` 接口。

### 风险 2：页面变成普通聊天 Demo

应对方案：

- 首页必须明确展示个人定位。
- 项目区必须放在主要页面结构中。
- 项目截图和视频必须作为重点内容展示。

### 风险 3：项目说服力不足

应对方案：

- 每个项目至少准备 1 张截图。
- 核心项目准备 30-60 秒演示视频。
- 项目详情必须说明个人职责和技术难点。

### 风险 4：AI 回答不准确

应对方案：

- 在 Dify 中维护准确的简历知识库。
- Prompt 中明确要求基于真实简历回答。
- 对未知信息要求 AI 明确说明“不确定”。

### 风险 5：开发范围过大

应对方案：

- MVP 只做单页作品集。
- 暂不做登录、后台、小程序、多语言。
- 先上线核心体验，再逐步迭代。

## 20. 结论

本产品的核心不是“做一个聊天框”，而是通过 AI 对话能力重新包装个人简历和项目作品集。

MVP 阶段应优先完成“个人定位 + AI 简历助手 + 项目证据 + 简历下载 + 后台内容维护”这条主线。只要这条主线清晰，项目就能同时服务于求职展示、面试讲解、技术能力证明和后续内容迭代。
